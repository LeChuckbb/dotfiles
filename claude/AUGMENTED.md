# Tidy First (Kent Beck)

## TIDY FIRST 접근법

모든 변경사항을 두 가지 타입으로 분리:

1. **구조적 변경(STRUCTURAL)**: 동작 변경 없이 코드 재배치 (이름 변경, 메서드 추출, 코드 이동)
2. **행동적 변경(BEHAVIORAL)**: 실제 기능 추가 또는 수정

- 구조적 변경과 행동적 변경을 **절대 같은 커밋에 섞지 말 것**
- 둘 다 필요한 경우 항상 구조적 변경을 먼저 할 것
- 구조적 변경 전후로 테스트를 실행하여 동작 불변을 검증할 것

## 커밋 규칙

다음 조건이 모두 충족될 때만 커밋:

1. 모든 테스트 통과
2. 모든 컴파일러/린터 경고 해결
3. 단일 논리적 작업 단위
4. 커밋 메시지에 structural/behavioral 명시

크고 드문 커밋보다 작고 빈번한 커밋 선호.
