# Blog Draft Review

블로그 초고를 사용자의 문체에 맞게 교정하고 피드백을 제공한다.

## 워크플로우

### 1단계: 초고 파일 확인

인자로 받은 파일 경로를 확인한다.
- 파일 경로가 제공되지 않았으면 사용자에게 요청
- 파일명만 주어진 경우 `blog-content/` 하위에서 탐색:
  - `/Users/mac/Library/Mobile Documents/iCloud~md~obsidian/Documents/Obsidian Vault/blog-content/`

### 2단계: published 글 읽기 및 문체 분석

published 폴더의 모든 .md 파일을 읽어 문체를 분석한다.

```
/Users/mac/Library/Mobile Documents/iCloud~md~obsidian/Documents/Obsidian Vault/blog-content/published/
```

**7개 이하로 읽지 않는다. 유형별 최소 3개, 총 15개 이상을 반드시 읽는다.**

구체적 절차:
1. Glob으로 published 폴더의 전체 .md 파일 목록 확보
2. 각 파일의 frontmatter(tags)와 제목으로 유형 분류: 서평, 기술 글, 회고, 에세이
3. 유형별로 3~5개씩 골라 총 15~20개를 읽는다
   - 선정 기준: 최신 글 우선 + 유형 내 다양성 확보 (짧은 글/긴 글 혼합)
4. **병렬 에이전트 활용**: 유형별로 에이전트를 나눠 동시에 읽기
5. 각 에이전트가 자기 유형의 문체 특징을 요약 → 통합하여 문체 프로파일 완성

분석 항목:
- **톤**: 격식체/구어체 비율, 독자와의 거리감
- **문장 구조**: 평균 문장 길이, 단문/복문 비율, 문단 길이
- **도입 패턴**: 질문형 / 상황 설정 / 인용 / 선언형
- **마무리 패턴**: 요약 / 여운 / 질문 / 행동 유도
- **글 유형별 구조**: 서평 / 기술 글 / 회고 / 에세이
- **인용·비유 활용**: 외부 인용 빈도, 개인 경험 비유 방식
- **frontmatter 형식**: tags 패턴, 필드 구성

분석 결과를 **문체 프로파일**로 정리한다.

### 3단계: 초고 읽기

인자로 받은 파일 경로의 초고를 읽는다.

### 4단계: 글 유형 판별

frontmatter의 tags와 본문 구조로 글 유형을 판별한다:
- **서평**: 책 이름, 저자 언급, 독후감 구조
- **기술 글**: 코드 블록, 설명-예제-결론 구조
- **회고**: 시간 순서, 감정 서사, 교훈 도출
- **에세이**: 주제 탐구, 개인 관점, 열린 결말

### 5단계: 교정 + 피드백 통합 제공

문체 프로파일 기준으로 아래를 함께 제공한다.

#### 피드백
- **전체 인상**: 글의 핵심 메시지와 독자에게 남기는 인상
- **강점**: 잘 된 부분을 원문 인용과 함께 구체적으로 명시
- **개선 방향**: 글 유형에 맞는 구조적 제안

#### 교정 제안
- **문체 일관성**: published 글과 어긋나는 톤, 어휘 선택
- **도입/마무리**: 패턴 일치도 및 개선안
- **문장 리듬**: 단조로운 구조, 너무 긴 문장
- **frontmatter 검증**: tags, 필드 누락 여부
- **구조 일치도**: 해당 글 유형의 일반적 구조와 비교

#### 원칙
- 원문 톤을 살리며 **최소한의 수정**만 제안
- 수정안은 Before/After 형식으로 제시
- **사용자 승인 후** 파일에 반영
- 쉼표(,)는 꼭 필요할 때만 사용한다. 나열·접속·삽입 등 문법적으로 필요한 경우에만 쓰고, 불필요한 쉼표를 남발하지 않는다.

## 공통 규칙

- 모든 출력은 **한국어**
- obsidian-vault-context 스킬은 참조하지 않는다 (블로그 교정은 볼트 구조와 무관)
- 파일 수정은 항상 사용자 확인 후 진행
